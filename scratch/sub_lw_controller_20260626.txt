## 2. Official LW Best Practices - Data Loading & Streaming Updates (update() vs setData)

**Core Rec from Docs** (https://tradingview.github.io/lightweight-charts/docs "Setting and updating a data"):
- **Initial preload (full historical slice, normalized)**: `series.setData(fullArray); chart.timeScale().fitContent();`
  Example (candlestick):
  ```ts
  const candlestickSeries = chart.addSeries(CandlestickSeries, { upColor: '#26a69a', ... });
  candlestickSeries.setData([
    { time: '2018-12-22', open: 75.16, high: 82.84, low: 36.16, close: 45.72 },
    // ... ~300 normalized bars, time as 'YYYY-MM-DD' or unix ts or BusinessDay obj
  ]);
  chart.timeScale().fitContent();
  ```
  "You can also use `setData` to replace all data items."

- **Streaming / deltas / appends / replay advance (preferred)**: `series.update(bar);`
  - Updates last bar OR appends new one if time > last.
  - For real-time/historical replay: call on new bar arrival or timer tick.
  Example:
  ```ts
  // Update most recent
  areaSeries.update({ time: '2018-12-31', value: 25 });
  candlestickSeries.update({ time: '2018-12-31', open: 109.87, high: 114.69, low: 85.66, close: 112 });
  // New bar (advance T)
  candlestickSeries.update({ time: '2019-01-01', open: 112, high: 112, low: 100, close: 101 });
  ```
- **Strong warning**: "We do not recommend calling `setData` to update the chart, as this method replaces all series data and can significantly affect the performance."
  - Matches task pitfalls exactly: use update() for deltas/stream; setData only init or rare full replace.
- Time formats: Consistent strings ('YYYY-MM-DD'), numbers (unix), or objects. LW v5+ handles well. Preload identical normalized slices for multi-symbol.
- Perf: Trivial for ~300 bars (engineered for thousands + frequent updates). v5.1+ data conflation helps.
- Sources: Official getting-started, series-types (setData examples for all types: CandlestickData, etc.).

**Mapping**: Arch/task-007 "realtime `series.update()` for new bars" + "WS client consume deltas". Preload from task-006 arenas (norm to 100). For replay: on WS new-bar-delta or local T-advance, update().

## 3. Multi-Chart Sync with Logical Ranges (Shared T / Scrub / FF / Pause)

**Key API**: ITimeScaleApi - `subscribeVisibleLogicalRangeChange`, `setVisibleLogicalRange`, `getVisibleLogicalRange`, `fitContent`.

**Official Sync Pattern** (https://tradingview.github.io/lightweight-charts/tutorials/how_to/set-crosshair-position - "Syncing two charts"):
```ts
// After creating chart1, chart2 + series + setData
chart1.timeScale().subscribeVisibleLogicalRangeChange(timeRange => {
  chart2.timeScale().setVisibleLogicalRange(timeRange);
});
chart2.timeScale().subscribeVisibleLogicalRangeChange(timeRange => {
  chart1.timeScale().setVisibleLogicalRange(timeRange);
});
// Also crosshair sync via subscribeCrosshairMove + setCrosshairPosition
```
- Logical range: `{ from: number, to: number }` (bar indices, fractional ok e.g. 5.2 = 5th bar + 20% visible). Starts from first data (neg before). Ideal for index-driven shared T (scrub to exact bar, independent of actual dates).
- Data range (time-based `setVisibleRange`) clips; logical allows margins/extrapolation control.
- TimeScale doc (https://tradingview.github.io/lightweight-charts/docs/time-scale): "The [setVisibleLogicalRange] method allows you to specify the visible range beyond the bounds of the available data. ... logical range represents ... indices". Use for "chart margin", align series. `fitContent()` for initial.
- GitHub/Discussions (e.g. #1163 multiple windows, #1108 React panes sync): Logical + subs work; for many charts "very slow if cross synched" -> master/slave (one drives others). Bidirectional risks loops.

**Best Practice for React Multi-Chart (Master/Slave to Avoid Loops)**:
- Designate "master" chart (e.g. primary instrument or controlled by ReplayController).
- Master sub -> set on slaves (no reverse sub, or guard with flag).
- On shared T change (WS or controller): compute logical range around current bar index (e.g. `{from: currentT - offset, to: currentT + visibleWidth}`), setVisibleLogicalRange on master (propagates).
- User scrub/pan/zoom on any: if master, update shared state/T visual; slaves follow.
- For FF/pause/scrub: controller adjusts visible logical + (optionally) highlights current T bar (e.g. via vertical line plugin or marker).
- Preload identical time-range data across charts.
- Also sync crosshair for "duel" feel.

**Pitfall - Sync Loops**: Bidirectional subs without master or `if (syncing) return; syncing=true; set...; syncing=false;` -> stack overflow or jitter. Official example is simple 2-chart; scale with care. Discussions note perf for 4+.

**Mapping to arch/task**: "timeScale logical sync for shared clock/scrub/FF". Server T drives (broadcast delta or currentT), client sets logical. Logical for "index-driven scrub" (prior sub).

## 4. Overlays: Markers (Orders/Events), PriceLines (SL/TP), Equity Series

**Markers** (https://tradingview.github.io/lightweight-charts/tutorials/how_to/series-markers):
- `createSeriesMarkers(series, markers)` (preferred/newer) or `series.setMarkers(markersArray)`.
- Full replace: keep array of all active markers, reconstruct + set on change.
- Example:
  ```ts
  const markers = [
    { time: '2018-12-23', position: 'aboveBar', color: '#f68410', shape: 'circle', text: 'Buy' },
    { time: '2018-12-25', position: 'belowBar', color: '#ef5350', shape: 'arrowDown', text: 'SL Hit' },
    // ...
  ];
  createSeriesMarkers(candlestickSeries, markers);  // or series.setMarkers(markers)
  ```
- Position: 'aboveBar'/'belowBar'/'inBar'. Shape: circle/square/arrowUp/etc. Text for labels (entries "L 10u @102", exits).
- Use for: Long/Short entries, fills, SL/TP triggers, sabotage events, news at specific T. Update on WS order/fill deltas or T-advance.
- Tip (community): To append, reconstruct full array from prior + new: `series.setMarkers([...existing, newMarker])`.

**Price Lines** (https://tradingview.github.io/lightweight-charts/tutorials/how_to/price-line + API):
- `const pl = series.createPriceLine({ price: 1234, color: '#3179F5', lineWidth: 2, lineStyle: LineStyle.Dashed, axisLabelVisible: true, title: 'SL 1.05' });`
- Update: `pl.applyOptions({ price: newPrice, title: '...' });`
- Remove: `series.removePriceLine(pl);` (keep refs in map by orderId).
- Disable built-in: `series.applyOptions({ lastValueVisible: false, priceLineVisible: false });`
- Perfect for live SL/TP levels (horizontal, labeled). Multiple per series (or per-instrument chart). Tie to active orders/positions from WS state.

**Equity / PnL Overlay**:
- Add parallel series on same chart or dedicated: `const equitySeries = chart.addSeries(LineSeries, { color: '#00ff00', lineWidth: 2, priceScaleId: 'right' /* or separate */ });`
- `equitySeries.setData([{time: '2018-12-22', value: 100}, ...]);` then `equitySeries.update({time: current, value: currentEquity});`
- Or Area for filled curve. Separate price scale to not distort main price. Update on every equity delta from WS (or computed locally from positions + bars, but server truth preferred).
- Prior sub/arch: "Equity as Line/Area series (parallel or separate scale)".

**General**: Attach to main price series (candlestick for instruments). Update reactively on deltas. For multi-instr: one chart per symbol or panes? (LW panes for volume/indicators within one; separate chart instances for distinct symbols common for sync).

**Mapping**: Task-007 "orders (market/limit/stop + SL/TP)", "equity curve mini-chart". Mech: orders/positions at T, PnL marked on T advance. Visual psych layer (sabotage hits as markers).

## 5. React/TS Lifecycle, Refs, Composability (Official Advanced Patterns)

**Official Advanced React Tutorial** (https://tradingview.github.io/lightweight-charts/tutorials/react/advanced - strongly recommended):
- Challenges: Hooks run bottom-up on mount / top-down cleanup -> parent/child order issues for chart/series.
- Solution: Refs + Context + forwardRef + useImperativeHandle + useLayoutEffect (for DOM timing).
- Structure:
  - `<Chart>` (container ref) > `<ChartContainer ref=...>` (creates chart via useRef internal, exposes api) > `<Series ref=seriesRef type=... data=... />` (creates series on chart context, setData, cleanup removeSeries).
  - Internal refs store methods for create/free.
  - Expose via useImperativeHandle so parent/WS can do `seriesRef.current.update(bar)`.
- Simple pattern (common in community):
  ```tsx
  const containerRef = useRef<HTMLDivElement>(null);
  const chartRef = useRef<IChartApi | null>(null);
  const seriesRef = useRef<ISeriesApi<'Candlestick'> | null>(null);  // or map for multi

  useEffect(() => {
    if (!containerRef.current) return;
    const chart = createChart(containerRef.current, { width: 600, height: 300, ...opts });
    chartRef.current = chart;
    const series = chart.addSeries(CandlestickSeries, { ... });
    seriesRef.current = series;
    series.setData(initialData);
    chart.timeScale().fitContent();

    const handleResize = () => { chart.resize(...); };
    window.addEventListener('resize', handleResize);

    return () => {
      window.removeEventListener('resize', handleResize);
      chart.remove();  // critical
      chartRef.current = null;
      seriesRef.current = null;
    };
  }, [/* deps */]);

  // Later (WS or timer)
  const handleDelta = (bar) => {
    if (seriesRef.current) seriesRef.current.update(bar);
  };
  ```
- Lifecycle notes: useLayoutEffect for create/apply (before paint). Cleanup subscriptions (e.g. `unsubscribeVisibleLogicalRangeChange`). Refs survive renders (store instances). Avoid reading .current in deps (stale). For conditional series: use state to mount <Series>, manage in refs.
- Advanced: Context.Provider for chart api down to children. useCallback for container. Theme/layout updates via applyOptions in effects.
- Resize: Always handle (container clientWidth/Height + observer or listener). LW canvas.
- TS: Native declarations included. Types: IChartApi, ISeriesApi, etc. from 'lightweight-charts'.
- Pitfalls: Double-mount in React StrictMode (dev); leaks if no cleanup; stale closures (use refs or latest-callback patterns).

**Mapping**: Arch "React: refs for live updates from WS/timer". Task-007 "reusable components". Official tutorial directly supports composable <Chart><Series ref=...> for multi-instr + overlays.

## 6. ReplayController / Timer Scaled by R + Scrub/FF/Pause

**No official "ReplayController"** - custom pattern (prior sub/arch explicitly suggest implementing one).
- **Timer pattern** (inspired by advanced React tutorial streaming useEffect + setInterval example):
  ```ts
  // In component or custom hook
  const replayControllerRef = useRef<{ timer: number | null; currentT: number }>({ timer: null, currentT: 0 });

  useEffect(() => {
    if (!isPlaying || R <= 0) {
      if (replayControllerRef.current.timer) clearInterval(...);
      return;
    }
    const baseMs = 1000;  // 1s real ~ 1 bar at R=1
    const interval = Math.max(50, baseMs / R);  // scale by R (FF faster = shorter interval)
    replayControllerRef.current.timer = setInterval(() => {
      // Option A (client-driven preview, reconcile server): advance local index, setVisibleLogicalRange, update local overlays
      // Option B (server-driven): request/emit "advance" or just wait for WS T-delta
      // On tick or T update:
      const newLogical = { from: currentT - 10, to: currentT + 20 };  // or focus current bar
      masterChart.timeScale().setVisibleLogicalRange(newLogical);
      // Update equity/markers if needed
    }, interval);
    return () => { if (timer) clearInterval(timer); };
  }, [isPlaying, R /*, currentT from WS */]);
  ```
- **Logical for scrub/FF/Pause**:
  - Scrub (user drag or slider on T): `chart.timeScale().setVisibleLogicalRange({ from: targetT - w, to: targetT + w });` (or set crosshair). Broadcast or send action if needed.
  - FF/Pause: UI buttons send TB actions (task-005/007); on R change from WS state, restart scaled timer or adjust visible follow.
  - Shared T: WS delta {T: number, R: number} drives visible follow (auto-scroll to latest or user-controlled offset). Logical ideal vs time-range (date-independent).
  - Pause (R=0): Stop timer, freeze visible or allow manual scrub.
  - Server auth (arch): Client timer for smooth UX/smoothing; server T authoritative (snap/reconcile on deltas). Conservative smoothing for variable R/latency.
- fitContent on init or after big jumps. Subscriptions for user-initiated pan/zoom feedback.
- Edge: Fractional T (smooth advance); non-trading days skipped in data.

**Mapping**: Prior sub "ReplayController with timer (scale interval by R/FF). ... Logical ranges for index-driven scrub." Arch "server T or R-scaled timer". Mech: R contested, T shared bar index. Task-007 TB controls drive/send actions, reflect R.

## 8. Key Pitfalls & Mitigations (Explicit in Prior Sub + Research)

- **Delta vs setData**: setData = full replace, perf killer + visible jumps/shifts/resets zoom (issues #1875, SO threads). Always update() for stream/replay appends. setData only init or full reload.
- **Sync loops**: Bidirectional logical subs -> loops/jitter. Mitigate: master/slave (one sub drives others), guard flags, or controlled (server T drives all via one path). Test with 2+ charts.
- **React refs/lifecycle**: Stale refs in timers/handlers; leaks (no unsubscribe/remove); StrictMode double; bottom-up hook issues (use official advanced ref+context pattern). Cleanup always: remove chart, unsub timeScale, clearInterval, null refs.
- **Visible jumps / range issues**: setData or bad logical resets view. Use logical for scrub (allows beyond data). No extrapolation on time-range (clips). fitContent carefully.
- **Markers**: No incremental; full setMarkers/createSeriesMarkers(array) each time. Keep master list.
- **Time consistency**: Uniform format across preload/deltas/WS. Handle fractional T for smooth.
- **Perf/scale**: Trivial here (~300 bars), but throttle updates, batch markers/priceLines. Resize listener always.
- **Other (from issues/docs)**: Insert middle data -> rebuild+setData. PriceLines atop series (z-order). Attribution per license. Mobile touch for scrub.
- **Desync with server**: Client timer optimistic; always snap to authoritative T/R/equity on deltas. Test lag (arch pitfalls).
